## 一、叙事状态机

### 基本模型

每个脚本是一个由节点组成的**有向无环图（DAG）**。节点分四种类型：

| 节点类型 | 说明 | 例子 |
|---------|------|------|
| `dialogue` | 纯对话，播完自动跳到下一节点 | 客人进场台词 |
| `choice` | 弹出 A/B/C 选项，玩家选后跳转 | "你怎么回应他的试探？" |
| `drink` | 弹出酒品选择，判定正确/错误后跳转 | "给他调什么？" |
| `narration` | 旁白/环境描写，通常用于场景转场和收尾 | 打烊后的叙述 |

### 节点结构（数据契约）

```typescript
interface ScriptNode {
  id: string;                    // 唯一标识，如 "dlg_01_01", "ch_01_01", "drink_01_01"
  type: "dialogue" | "choice" | "drink" | "narration";
  
  // dialogue / narration 节点
  lines?: DialogueLine[];       // 对话行
  next?: string;                // 下一节点 ID（无选择时）
  
  // choice 节点
  choices?: {
    label: string;              // 选项显示文字
    next: string;               // 选中后跳转节点 ID
    effects?: StateEffect[];    // 选择后果
    condition?: Condition;      // 选项可见条件（可选）
  }[];
  
  // drink 节点
  available?: string[];         // 可选酒品
  correct?: string;             // 正确酒品
  correctNext?: string;         // 正确跳转
  wrongNext?: Record<string, string>; // 各错误酒品对应跳转
  effects?: Record<string, StateEffect[]>; // 选择后果
}

interface DialogueLine {
  speaker: string;              // 说话人，或 "narration"
  text: string;                 // 台词
  emotion?: string;             // 表情/语气标签（可选）
}

interface StateEffect {
  target: string;               // 状态变量路径，如 "lendro.trust", "cat.affinity"
  operation: "add" | "set" | "multiply";
  value: number;
}

interface Condition {
  variable: string;             // 状态变量路径
  operator: ">=" | "<=" | "==" | "!=" | ">" | "<";
  value: number | boolean | string;
}
```

### 运行流程

```
加载脚本 → 从 entry 节点开始 → 
  如果是 dialogue: 逐行播放 → 跳到 next
  如果是 choice: 显示选项 → 等待玩家选择 → 跳到对应 next → 应用 effects
  如果是 drink: 显示酒品选择 → 等待玩家选择 → 判定正确/错误 → 跳到对应分支 → 应用 effects
  如果是 narration: 播放 → 跳到 next
→ 直到 next 为空或 "END"
```

### 指令层（舞台指令）

脚本中的 `[BG ...]`、`[BGM ...]`、`[SE ...]` 等指令需要在对话流中解析并触发：

| 指令 | 含义 | 后端操作 |
|------|------|---------|
| `[BG id]` | 切换背景 | 触发前端背景切换事件 |
| `[BGM id]` | 切换音乐 | 触发前端音乐切换事件 |
| `[LIGHT ...]` | 灯光变化 | 触发前端光照效果 |
| `[SE id]` | 音效 | 触发前端音效播放 |
| `[ENTER sprite_id]` | 角色入场 | 触发前端角色 sprite 显示 |
| `[EXIT sprite_id]` | 角色离场 | 触发前端角色 sprite 隐藏 |
| `[PAUSE seconds]` | 停顿 | 等待指定秒数 |
| `[CHOICE id]` | 选择节点标识 | 标记选择开始位置 |
| `[GOTO id]` | 跳转到节点 | 跳转到指定节点 |
| `[END]` | 脚本结束 | 触发打烊流程 |

## 二、选择系统

### 基本规则

- 每个选择节点有 3 个选项（个别场景可能有 2-4 个）
- 选择立即生效，不可撤销
- 部分选项有**可见条件**（比如"只有赌徒 trust ≥ 50 时才能选告诉他死神在酒馆"）
- 未满足条件的选项直接隐藏，而不是灰掉

### 选项影响

| 影响类型 | 说明 | 例子 |
|---------|------|------|
| 好感度变化 | trust/affinity/openness ± N | "递烟" → lendro.trust +4 |
| 信息揭示 | 某些选项会触发额外对话/背景故事 | "那是图纸？" → 维修师讲出理想 |
| flag 设置 | 设置布尔旗标，影响后续剧情 | "告诉死神" → gambler_betrayed = true |
| 结局导向 | 累积影响最终结局走向 | 猫女亲和度 ≥ 一定值 → 结局 A |

### 核心选择清单（Demo 中的关键分支）

| 天数 | 选择节点 | 选项 | 影响 |
|------|---------|------|------|
| Day 1 | ch_01_01 回应盘查 | A:冷淡 / B:直白 / C:反问 | 影响警探 trust 初始值 |
| Day 1 | drink_01_01 | 白酒(正确)/威士忌/啤酒/柠檬水 | trust +5 或不变 |
| Day 1 | ch_01_04 临别 | A:危险 / B:扔掉 / C:倒酒压惊 | trust 不同程度变化 |
| Day 2 | ch_02_01 接客 | A:冷淡 / B:多看一眼 / C:指出机油 | 影响维修师 openness |
| Day 2 | drink_02_01 | 啤酒(正确)/威士忌/白酒/其他 | openness +5 或不变 |
| Day 2 | ch_02_02 是否搭话 | A:沉默 / B:问图纸 / C:戳穿身份 | 决定 blueprint_revealed 时机 |
| Day 2 | ch_02_cat 怎么对猫 | A:摸摸 / B:给啤酒 | 猫亲和度 +3 或 +1 |

## 三、调酒系统

### 机制

玩家从可选酒品中选择，后端判定正确/错误：

- **正确** → 播放正确对话 → trust/openness +5（或其他设计值）
- **错误** → 客人嫌弃 → 部分角色会要求换酒（自动给正确酒）+ 信任不增加（或微降）
- **错误不会卡关。** 故事必定继续。

### Demo 调酒链

| 角色 | Day | 正确酒 | 可选范围 | 备注 |
|------|-----|--------|---------|------|
| 警探 | 1 | 白酒 | 白酒/威士忌/啤酒/柠檬水 | 烈、纯、不花哨 |
| 警探 | 5 | 威士忌 | 威士忌/白兰地/啤酒 | 经典，建立信任后 |
| 维修师 | 2 | 啤酒 | 啤酒/威士忌/白酒/调酒 | 永远是啤酒 |
| 维修师 | 7 | 啤酒 | 啤酒/威士忌/白酒 | "老样子" |
| 牛仔 | 3 | 烈酒(白酒) | 白酒/威士忌/啤酒 | 强悍外壳 |
| 牛仔 | 8 | 烈酒 | — | 震惊中，延续口味 |
| 牛仔 | 10 | 露酒 | — | 与猫女和解后喝同一种 |
| 赌徒 | 4 | 柠檬水 | 柠檬水/白酒/啤酒/威士忌 | 不敢喝酒 |
| 赌徒 | 6 | 威士忌 | 威士忌/白酒/啤酒 | 放松信任 |
| 赌徒 | 9 | 白酒(暖/选择的碰杯)或葡萄酒(冷/认命的碰杯) | 取决于是否告诉死神 | 导向不同结局 |
| 死神 | 9 | 露酒 | 露酒/果酒/威士忌 | 同场出现 |

### 调酒判定逻辑

```typescript
function evaluateDrink(drinkNode: DrinkNode, playerChoice: string): DrinkResult {
  if (playerChoice === drinkNode.correct) {
    return {
      correct: true,
      nextNode: drinkNode.correctNext,
      effects: drinkNode.effects?.["correct"] || [],
      dialogue: drinkNode.correctDialogue
    };
  }
  return {
    correct: false,
    nextNode: drinkNode.wrongNext?.[playerChoice] || drinkNode.wrongDefaultNext,
    effects: drinkNode.effects?.[playerChoice] || [],
    dialogue: drinkNode.wrongDialogues?.[playerChoice]
  };
}
```

## 四、好感度/状态追踪

### 状态变量命名规范

```
{角色id}.{变量名}

例：
  lendro.trust          // 警探信任度
  enneer.openness       // 维修师敞开心扉程度
  cat.affinity          // 猫女亲和度
  aletor.relaxation     // 赌徒放松程度
  cowgirl.defense       // 牛仔防御值（越低越真实）
```

### 需要持久化的状态

每个角色至少追踪：

| 字段 | 类型 | 说明 |
|------|------|------|
| `trust` / `openness` / `affinity` | number (0-100) | 好感度主值 |
| `visits` | number | 已来访次数 |
| `current_stage` | string | 当前阶段（stage_1/stage_2/...） |
| `flags` | Record<string, boolean> | 剧情旗标集合 |

全局状态：

| 字段 | 类型 | 说明 |
|------|------|------|
| `current_day` | number (1-11) | 当前天数 |
| `completed_days` | number[] | 已完成的天数列表 |
| `cat_unlocked` | boolean | 猫是否已签约入住 |
| `cat_affinity` | number (0-100) | 猫的全局亲和度 |
| `gambler_betrayed` | boolean | 是否告诉了死神（Day 9 决定） |
| `cowgirl_catwoman_resolution` | "reconcile" \| "break" \| null | 大纷争结果 |

### 状态变化触发时机

每次选择或调酒判定后，后端必须：
1. 应用该节点的 `effects` 列表
2. 检查是否有跨天解锁条件被满足
3. 更新 `visits` 计数（每次客人离场时 +1）

## 五、天推进系统

### 推进逻辑

```
当前天脚本执行完毕（遇到 [END]）
  → 触发打烊场景（旁白 + 猫互动可选）
  → 检查下一天解锁条件
  → 如果满足 → 解锁下一天
  → 如果还有条件未满足 → 下一天灰掉/提示
  → 玩家手动点"下一夜"进入新天
```

### 解锁条件

| Day | 解锁条件 |
|-----|---------|
| Day 2 | Day 1 完成 |
| Day 3 | Day 2 完成 |
| Day 4 | Day 3 完成 |
| Day 5 | Day 4 完成 |
| Day 6 | Day 5 完成 |
| Day 7 | Day 6 完成 |
| Day 8 | Day 7 完成 **且** cat.affinity ≥ 5（确保玩家至少喂过猫） |
| Day 9 | Day 8 完成 **且** aletor.visits ≥ 2 |
| Day 10 | Day 9 完成 **且** cowgirl.visits ≥ 2 **且** cat.affinity ≥ 8 |
| Day 11 | Day 10 完成（无额外条件，全角色故事完成后自动解锁） |

> Day 8 的亲和度门槛很低（摸一次猫就够了），主要是防止"完全没理猫但猫突然变成灵体"的逻辑断裂。

### 打烊流程

每天脚本执行完毕后的固定流程：

```
1. [EXIT 当前客人]（如果还没退场）
2. 播放旁白（narr 节点）
3. 猫互动机会（Day 2 起）：玩家可选择投喂
4. [BG tavern_night_empty] 空酒馆画面
5. 显示"下一夜"按钮 / 自动过渡到新天
```

## 七、存档系统

### 需要保存的数据

```typescript
interface SaveData {
  version: string;
  timestamp: number;
  currentDay: number;
  completedDays: number[];
  
  // 角色状态
  characters: {
    [characterId: string]: {
      visits: number;
      trust: number;
      openness: number;
      affinity: number;
      stage: string;
      flags: Record<string, boolean>;
    };
  };
  
  // 全局旗标
  flags: {
    cat_unlocked: boolean;
    gambler_betrayed: boolean | null;       // null = 还没到 Day 9
    cowgirl_catwoman_resolution: "reconcile" | "break" | null;
  };
  
  // 当前脚本执行位置（如果在中途存档）
  scriptState: {
    day: number;
    currentNodeId: string;
    dialogueIndex: number;
  } | null;
}
```

### 存档时机

- 每天结束（打烊后）**自动存档**
- 支持手动存档（任意时刻）
- 单存档位即可（Demo 不需要多档位）

## 九、开发建议

### 第一步：搭架子

先把状态机跑通。用一个最简单的脚本（比如 Day 1 的前三个节点）验证：
- 脚本解析 → 节点推进 → 对话播放 → 选择弹出 → 跳转正常

### 第二步：调酒系统

独立模块。输入可选酒品列表和正确答案，返回判定结果和对话。

### 第三步：状态管理

实现 Zustand store（前端已有 `useGameStore.ts`），对接状态变化。

### 第四步：逐天接入

按 Day 1 → Day 2 → ... 的顺序，每天写一个脚本、接入一个。不要一次写完全部再调试。

### 优先级

| 优先级 | 内容 |
|--------|------|
| P0 | 状态机 + 选择系统 + 调酒判定 |
| P1 | 场景切换 + BGM 切换 |
| P2 | 猫互动 + 投喂系统 |
| P3 | 存档系统 |
| P4 | 复杂特效（结霜、紫光、混沌叠加态） |
